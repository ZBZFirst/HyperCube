# Technical Article: HyperCube Literature Screening Pipeline Architecture

## Abstract

This article describes the software architecture of the HyperCube literature-screening pipeline and explains why its implementation choices are appropriate for the current problem. The system joins a JavaScript-based PubMed retrieval layer to a Node ECMAScript module that performs row-wise eligibility screening through an OpenAI-compatible LM Studio endpoint. The core architectural idea is to use a normalized CSV file as the stable handoff boundary between retrieval and inference. That boundary isolates acquisition logic from screening logic, preserves provenance, and allows prompts, models, presets, and execution locations to change without invalidating the dataset. The screening client is implemented in `screenPubMedCsv.mjs` rather than Python because the repository is already JavaScript-centered, Node is readily deployable on the remote LM Studio host, and the task is orchestration rather than scientific computing. In the present system, reproducible artifact generation depends on the interaction of the loaded Qwen model, the published LM Studio prompt preset, strict JSON schema output, and deterministic decoding with `temperature: 0`. LM Studio is important here not only as a host for model loading, but as the control surface that makes offline reproducibility possible by exposing inference, model, input, output, and variable controls that can be shared and re-used across runs. Results from the current run show reliable schema-constrained output generation and deterministic CSV augmentation. The pipeline therefore demonstrates a small, reproducible, and maintainable architecture for local first-pass literature triage.

## PART I. QUESTION

The technical question addressed by this work is not simply how to call a language model from a script. The more important question is how to build a maintainable screening pipeline that can survive iteration in prompts, model selection, hardware placement, and research topic without becoming brittle. In this project, PubMed retrieval, data normalization, and local model screening must work together, but they should not be entangled.

HyperCube already existed as a JavaScript-based codebase for collecting and structuring PubMed output. That prior condition matters. Once article metadata, abstracts, identifiers, and search provenance are produced in JavaScript, the screening step can either remain in the same runtime family or introduce a second language boundary. A Python implementation was possible, but it would have added another operational layer without solving a problem that is unique to Python.

The architectural requirement was therefore to choose a screening implementation that meets five constraints. First, it must preserve the dataset as an inspectable artifact rather than burying logic inside transient prompts or notebook state. Second, it must support deterministic parsing so that row-level model outputs can be appended back into tabular form without manual cleanup. Third, it must run both from the coordinator machine and from the LM Studio host used for overnight batches. Fourth, it must be easy to audit and modify in the same repository as the retrieval code. Fifth, it must avoid unnecessary dependency weight because the problem is largely transport, validation, and file transformation.

These constraints led to the decision to treat the CSV as the contract and the screening step as a lightweight orchestration layer. The technical question then becomes: what implementation best preserves repository continuity, deployment simplicity, and reproducibility while calling a local language model endpoint and writing structured results back into the evidence table?

## PART II. METHODS

The pipeline has two main software stages. The first stage retrieves and normalizes PubMed records. HyperCube gathers title, abstract, authors, date, DOI, PMC references, keywords, MeSH terms, PubMed query provenance, and the research question supplied for the run. Those fields are written into a CSV that serves as the evidence table. This file is not an incidental export. It is the system boundary. Once the dataset exists in that form, the retrieval process is complete and the screening process can evolve independently.

The second stage is implemented in `screenPubMedCsv.mjs`, a Node ECMAScript module. That script reads the CSV, converts each row into a structured object, extracts `ResearchQuestion`, `Title`, and `Abstract`, and assembles an LM Studio request against `/v1/chat/completions`. The script also defines a system prompt and a strict JSON schema. The prompt tells the model to treat the research question as a restrictive eligibility criterion and to treat the title and abstract as the only evidence for inclusion. The JSON schema constrains the response to either `screened_article` or `invalid_input`, with required fields that can be written directly into appended CSV columns.

This design choice is central. The model is not permitted to return arbitrary prose that later code must interpret heuristically. Instead, the script enforces a machine-readable contract at inference time. The downstream effect is that `choices[0].message.content` can be parsed deterministically, mapped into screening headers, and serialized back into a second CSV. The output remains reviewable because the original article row and the model judgment remain side by side.

The inference layer should therefore be understood as a controlled variable set rather than as a single model-selection decision. In the tested configuration, the loaded model was `qwen2.5-3b-instruct`, specifically the GGUF artifact `qwen2.5-3b-instruct-q4_k_m.gguf`, served through LM Studio. The prompting approach was also not private or purely ad hoc: the related LM Studio preset configuration was published on the web as `https://lmstudio.ai/paulwasthere/research-assistant`, with local preset entries including `@lmstudio-hub:paulwasthere/research-assistant` and `@lmstudio-hub:paulwasthere/focused-research-assistant`. This matters because reproducibility in this system depends not only on naming the inference model, but also on preserving the prompt behavior and inference controls that shape the final artifact.

LM Studio’s role is therefore broader than simple model serving. It provides a practical offline environment in which model choice, prompt preset, endpoint behavior, decoding variables, and structured-output constraints can all be controlled together. That combination allows a configuration to be shared and re-applied instead of being rebuilt informally from memory. In this workflow, that means the screening behavior can be documented, published, imported, adjusted, and rerun locally while keeping article content and intermediate decisions under local control.

One of those controls is especially important: `temperature: 0`. In this pipeline, zero temperature is not a cosmetic preference. It is a methodological requirement for reliable artifact generation. Because the output is expected to conform to a strict schema and populate stable CSV columns, unnecessary sampling variance would work against the objective of deterministic row-wise screening. With `temperature: 0`, the model is pushed toward the most stable completion available under the prompt, schema, and evidence constraints. That does not make the system perfectly deterministic across all environments, but it materially reduces avoidable variability and is therefore paramount for this exercise.

The reason for using `.mjs` rather than Python is practical and architectural. HyperCube is already a JavaScript repository. Using Node keeps the retrieval and screening layers in the same language family, which lowers maintenance cost and keeps the pipeline understandable to anyone already reading the project’s source. In this case, the screening step is not a data-science workload that depends on NumPy, pandas, or specialized scientific libraries. It is primarily a sequence of operations: read file, parse CSV, construct JSON payload, send HTTP request, validate response, append fields, and write output. Node fits that workload directly.

Deployment considerations also favor the current choice. The remote LM Studio host already runs Node cleanly and can launch the batch through a plain Windows `.cmd` wrapper that calls `node.exe` and the screening module. That avoids provisioning a separate Python environment, reconciling interpreter versions, or documenting package installation just to execute a lightweight transport script. The gain is not that JavaScript is universally better than Python. The gain is that, in this repository and on this host topology, Node reduces environment friction while preserving behavior.

Finally, the script is designed to support both single-row sequential execution and limited concurrency. Command-line arguments allow the caller to override endpoint, model, output path, row limit, and concurrency. This makes the orchestration layer portable across LAN execution, remote-host execution, and future runs on stronger hardware.

## PART III. RESULTS

The present architecture was exercised on a real test dataset derived from a PubMed search concerning drones and sea measurement. The input file contained 27 structured rows. The screening stage submitted each row to a local-network LM Studio node serving `qwen2.5-3b-instruct` through an OpenAI-compatible endpoint. The resulting output file preserved the original article fields and added `LlmStatus`, `LlmFitForReview`, `LlmSummary256`, `LlmReason`, `LlmInvalidKind`, `LlmReceivedText`, `LlmModel`, `LlmEndpoint`, `LlmScreenedAt`, and `LlmError`.

From a technical standpoint, the most important result is reliability rather than semantic perfection. The run completed without transport failure. Each row generated schema-constrained output that could be written back into CSV form. No free-text post-processing layer was needed to recover valid fields. This confirms that the structured output contract is functioning as intended and that the parser-writer path is stable for the tested dataset. Just as importantly, it confirms that the artifact was produced by a coordinated set of variables working together: the selected Qwen model, the prompt preset logic, the restrictive system prompt, `response_format` with strict JSON schema, and `temperature: 0` decoding.

Timing data in the produced CSV indicate that the full run required roughly 312 seconds for 27 rows, or about 11.6 seconds per row. That is not high throughput by server standards, but it is entirely practical for a first-pass local screening pipeline on modest hardware. The remote host was not a dedicated AI server; it was a workstation-class machine with a Ryzen 5 5600G, integrated Radeon graphics, and about 33.7 GB of RAM. The significance of that result is that the architecture works under ordinary local conditions rather than requiring cloud infrastructure or a high-end GPU box.

The present prompt configuration marked all 27 rows as `screened_article` and all 27 as not fit for review. From a technical perspective, that outcome is informative. It shows that the transport layer, response schema, parser, and CSV writer all behaved consistently even when the screening logic itself remained overly strict. In other words, the architecture succeeded even where the current prompt needs refinement.

The pipeline also demonstrated a useful operational property: the same screening script can be run from the coordinator laptop against a LAN endpoint or moved onto the LM Studio host and pointed at `127.0.0.1` for overnight batch work. That mobility is exactly what the architecture was intended to support. Because the handoff boundary is a CSV and the screening layer is a self-contained Node module, execution location can change without forcing redesign of the data layer.

## PART IV. DISCUSSION

The main technical contribution of this project is the combination of a stable data contract and a lightweight screening client. The CSV boundary prevents retrieval logic from collapsing into screening logic. The structured output schema prevents model output from dissolving into unstructured notes. Together, those two choices make the system inspectable, testable, and easy to iterate.

That contribution would be described incompletely if the discussion stopped at model loading and pipeline construction. In practice, the final screened CSV is an artifact generated by interacting controls. The loaded model contributes capability. The published LM Studio preset contributes reusable prompt behavior. The system prompt contributes role restriction. The JSON schema contributes output shape. The endpoint configuration contributes execution locality. `Temperature: 0` contributes stability. Only the combination of those variables produces a row-wise artifact that is suitable for audit and comparison.

This is also where LM Studio contributes a distinct methodological advantage. Because it supports local inference with explicit control over model selection, inference settings, prompt presets, input structure, output structure, and adjustable variables, it enables offline reproducibility rather than merely offline execution. A researcher can keep human review in the loop while still accelerating batch processing. The model does not replace the reviewer; it makes the repetitive first-pass work faster and more consistent. At the same time, the shared preset and controlled endpoint settings make it possible to hand the configuration to another operator and have them reproduce substantially the same screening behavior on local hardware.

The Node `.mjs` choice should therefore be understood correctly. It is not an ideological statement about programming languages. It is an engineering decision about local fit. In a repository already centered on JavaScript, with a screening task that is mostly I/O orchestration, and with a remote host that can launch Node jobs directly, `.mjs` is a lower-friction and more coherent option than introducing Python solely for the screening layer. If the project later grows into heavy statistical analysis, advanced dataframe manipulation, or tight integration with Python-native research tooling, then a Python stage may become justified. At present, the current implementation aligns better with the architecture actually required.

The results also clarify where future work belongs. The transport path is adequate. The schema contract is adequate. The major weakness is prompt behavior and, secondarily, throughput. Prompt tuning should improve the distinction between off-topic rows and borderline relevant rows without weakening the rule that only explicit evidence from the title and abstract should count. Throughput can be improved by using the script’s concurrency options more aggressively on stronger hardware or by keeping batch execution local to the model host. Any such future changes should preserve the current emphasis on low-variance decoding. For this kind of structured screening system, raising temperature casually would undermine comparability across runs and weaken the reliability of the generated CSV artifact.

From a software engineering perspective, the system is already useful because it produces durable artifacts at every critical stage: a retrieval CSV, a screening script with explicit arguments, a prompt contract, a JSON schema, and an augmented output CSV. That is a stronger foundation than an ad hoc workflow based on copied prompts, manual browsing, or notebooks with hidden intermediate state.

For these reasons, the HyperCube screening pipeline should be considered a sound small-system architecture for early literature triage. It defends its code choices not by abstract preference, but by matching the implementation to the repository, the host environment, and the actual shape of the task.
